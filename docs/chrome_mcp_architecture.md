# Chrome Bridge Architecture

## Goal

Build a local Chrome extension plus Python bridge that gives the bot a browser-native control layer closer to a semantic MCP workflow:

- read structured page state
- list interactive elements with stable ids
- click by element id
- set form values directly
- scroll the page or a container
- keep per-tab state on the browser side

## Components

### 1. Chrome extension

Location:

- `chrome_mcp/extension/manifest.json`
- `chrome_mcp/extension/service_worker.js`
- `chrome_mcp/extension/content_script.js`

Responsibilities:

- connect to the local bridge over localhost
- collect semantic page state from the live DOM
- expose interactive elements with stable ids plus label-aware metadata
- execute direct DOM actions against resolved elements
- keep tab state discoverable to the Python side

### 2. Local Python bridge

Location:

- `job_apply_bot/chrome_mcp_server.py`

Responsibilities:

- accept extension connections
- track connected tabs and the latest snapshot for each tab
- expose a simple local RPC layer for tooling and later bot integration
- validate request/response payloads
- keep a small action log for debugging

### 3. Job bot integration

Initial phase:

- standalone CLI and bridge only

Next phase:

- optional browser-provider adapter in the apply flow that uses the Chrome bridge instead of Playwright for supported sites

## Message Model

Messages are plain JSON with a top-level `type`.

Core browser-to-bridge messages:

- `hello`
- `page.state`
- `tabs.list.result`
- `action.result`
- `pong`

Core bridge-to-browser messages:

- `ping`
- `tabs.list`
- `page.collect`
- `page.read`
- `page.find`
- `page.action`

## Snapshot Shape

Each page snapshot should include:

- `url`
- `title`
- `scrollX`
- `scrollY`
- `viewport`
- `visibleTextExcerpt`
- `interactiveElements`

Each element should include at least:

- `id`
- `tagName`
- `role`
- `label`
- `name`
- `type`
- `value`
- `placeholder`
- `disabled`
- `checked`
- `required`
- `visible`
- `rect`

Stable ids should be generated by the content script and refreshed when the DOM changes enough that the old handle is no longer safe.

## Initial Action Set

### `page.read`

Return the latest structured state for a tab. The Python bridge can also compact the response down to interactive elements only, which keeps controller context small.

### `page.find`

Search the interactive element list semantically by label, text, placeholder, name, or aria label. This is the closest local equivalent to Claude's `find`.

### `page.action`

Execute `click`, `setValue`, or `scroll` against a stable element id. The content script resolves the live element, scrolls it into view when needed, and dispatches the DOM events directly.

### `page.collect`

Return the full raw page state from the content script, including the long visible text excerpt and the full interactive element list.

## Scope For V1

Included:

- active-tab control
- one extension connection
- semantic snapshot extraction
- label-aware element metadata
- semantic `page.find`
- direct click/input/scroll actions
- local CLI access
- smoke-test documentation

Not included:

- screenshots through the bridge
- multi-window orchestration
- full MCP stdio compatibility
- native OS control
- production auth or encryption

## Integration Notes

- The bridge should stay optional and not break the current Playwright path.
- The protocol should be small and explicit so later migration to a true MCP server is straightforward.
- The extension should prefer DOM semantics over screenshots.
- The first release should optimize for inspection and reliability, not feature count.
- The validated local smoke path is Playwright Chromium with the unpacked extension loaded against `runs/chrome_mcp_fixture.html`.
